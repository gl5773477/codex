# Territory Sandbox 协作上下文

## 项目定位

- 这是一个网页版的“上帝视角”大陆演化沙盘原型。
- 当前目标不是一口气做成完整策略游戏，而是先验证一件事：
  - 看着大陆自行演化、国家自行扩张，这件事本身是否有趣。
- 项目的核心价值在于“可观察、可演化、可轻度干预、可快速迭代”。

## 当前阶段目标

- 优先把最核心的自动演化闭环做扎实。
- 保持项目始终处于可运行、可观察、可继续迭代的状态。
- 优先增强演化过程的观赏性、可读性和可控性。

## 当前核心玩法

1. 地图被划分为多个领地。
2. 领地归属于不同势力或国家。
3. 每块领地会自动生产人口。
4. 人口积累后会自动向相邻敌方领地扩张。
5. 领地被占领后，归属和颜色会发生变化。
6. 玩家以“上帝视角”观察整个大陆，并做少量干预。

## 玩家角色定位

- 玩家不是扮演单一国家。
- 玩家是整个世界的观察者与干预者。
- 当前版本的干预应保持轻量，优先包括：
  - 暂停 / 继续
  - 倍速切换
  - 点击查看领地信息
  - 给选中领地增加人口

## 范围约束

- 除非用户明确提出，否则不要主动引入复杂经济系统、建筑树、科技树、外交系统、英雄养成或复杂实时战斗。
- 除非用户明确提出，否则不要把当前项目重构成另一套全新架构。
- 当前阶段不追求“大而全”，优先追求“演化过程看起来有意思”。
- “小人”更适合作为人口或活跃度的视觉表达，而不是复杂实体单位。
- 当前继续使用六边形网格是可接受方案，地图真实感可以后续逐步增强。

## 现有实现基线

- 后续修改默认都基于当前 `territory-sandbox/` 内已有实现继续演进。
- 技术栈基线为：
  - React
  - TypeScript
  - Vite
  - Canvas 2D
- 当前主要模块职责：
  - `src/game/types.ts`：世界、领地、势力等类型定义
  - `src/game/constants.ts`：地图尺寸、势力颜色、模拟参数
  - `src/game/hex.ts`：六边形坐标、邻接、点击判定
  - `src/game/world.ts`：大陆 / 海洋生成、初始领地归属
  - `src/game/simulation.ts`：人口增长、自动扩张、占领、选中逻辑
  - `src/game/render.ts`：Canvas 地图绘制、人口点、数字信息
  - `src/App.tsx`：应用外壳、控制面板、神力按钮、交互

## 后续协作原则

- 默认采用“最小可验证改动”。
- 优先做能明显提升演化体验的改动，而不是先堆复杂系统。
- 当多个方案都可行时，优先选择更能提升以下三件事的方案：
  - 玩家是否看得懂世界正在发生什么
  - 演化节奏是否自然且有观赏性
  - 玩家是否能以较低成本观察和轻度干预世界
- 始终保留快速运行、快速观察、快速迭代的能力。

## 跨工具协作约定

- 用户可能会在 ChatGPT 与 Codex 之间切换推进同一个项目。
- 仓库中的当前文件状态始终比历史对话更可信，应默认以仓库现状为准。
- 每次开始新的修改前，先重新检查仓库当前内容与关键文件，确认是否已有新增实现、结构变化或需求落地。
- 如发现仓库内容与历史上下文不一致，应优先以最新代码为基线，再结合历史背景做判断。
- 未经明确要求，不要因为旧方案或旧对话而回退用户已经在仓库中完成的新改动。

## 优先演进方向

- 让大陆形状和国家边界更自然。
- 让扩张逻辑更聪明、更有戏剧性，但不要显著增加系统复杂度。
- 增加少量更有意思的上帝干预能力。
- 优化人口、战争与领地变化的可视化表现。
- 逐步加入事件、特殊人物等轻量层次，而不是一次性做完整管理系统。
